# ADR-003: Asynchronous HeyGen Video Pipeline with Graceful Degradation

## Status

Accepted

## Date

2026-04-13

## Context

Each course section requires a 1-3 minute talking-head video generated by the
HeyGen API. HeyGen video generation is asynchronous -- you submit a job and
poll for completion. Typical generation time is 2-5 minutes per video. An
8-section course would take 16-40 minutes if videos were generated sequentially.

The system must generate video without making the overall course generation
unacceptably slow, and must handle the reality that HeyGen jobs can fail,
time out, or be rate-limited.

We considered three strategies:

1. **Sequential video generation**: Generate each video one at a time, wait
   for completion before starting the next.
2. **Parallel submission, deferred collection**: Submit all video jobs at once,
   continue with HTML generation, then collect completed videos before final
   assembly.
3. **Background video generation**: Generate the HTML course immediately (text
   lecturer), then generate videos asynchronously and update the course later.

## Decision

We will use **Option 2: Parallel submission with deferred collection**.

The pipeline:
1. After synthesis (Phase 4), submit all section video jobs to HeyGen in
   parallel in a single batch.
2. Immediately proceed to HTML generation (Phase 6) -- do not wait for videos.
3. After HTML generation completes, enter the video collection phase: poll all
   jobs every 15 seconds until all complete or timeout (10 minutes per video).
4. Download completed videos, patch the HTML with video URLs, and write output.
5. For any video that failed or timed out, the HTML retains the text lecturer
   fallback for that section.

## Rationale

**Option 1 rejected** because sequential generation is unacceptably slow.
8 sections x 3 min average = 24 minutes of idle waiting. Users will not
tolerate this.

**Option 3 rejected** because it creates a two-step user experience. The user
gets a "done" notification but then has to wait for a second "videos ready"
notification. This is confusing and error-prone -- the user might share the
course before videos are ready, or close the terminal and lose the background
job.

**Option 2** is the optimal balance: all videos generate in parallel (wall clock
time = slowest single video, not sum of all videos), and the HTML generation
runs concurrently with the first few minutes of video generation, further
reducing total time. The user gets a single "done" notification when everything
is ready.

## Consequences

### Positive

- Total generation time with video: ~8-12 minutes (vs. ~24-40 min sequential)
- HTML generation is not blocked by video pipeline
- All-or-nothing delivery: user gets a complete artifact when notified
- Failed videos degrade gracefully -- the course is never broken

### Negative

- More complex orchestration (two parallel tracks + merge)
- Must handle partial completion states (some videos done, others failed)
- Polling loop consumes API calls (though these are lightweight GET requests)

### Mitigations

- The orchestrator uses a simple state machine per job: `submitted -> processing
  -> completed | failed | timeout`
- Polling interval of 15 seconds balances responsiveness vs. API load (4 calls/
  min per video, 32 calls/min for an 8-section course)
- 10-minute timeout per video is generous (HeyGen docs cite 2-5 min typical)
  but bounded to prevent indefinite hangs

## Fallback Behavior

| Video Status | Lecturer Pane Behavior |
|-------------|----------------------|
| `completed` | Video player with HeyGen video, transcript below |
| `failed` | Text lecturer (existing behavior), warning badge |
| `timeout` | Text lecturer (existing behavior), warning badge |
| All failed | Full text-only course, user-facing warning with suggestion to retry |

The fallback is seamless: the HTML always contains the text lecturer script.
The video player is an enhancement layered on top. If the video `src` fails
to load, the player hides and the transcript expands to fill the pane.

```
  |       |

  |       |

  |       |

  |       |
  |       |  [Download completed videos as they finish]
  |       |  [Mark failed/timed-out jobs]
  |       |

  +---> Assembly: patch HTML with video URLs
  |       |
  |       v
  +---> Final output written
```
